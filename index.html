1: How to define a measure

DON'T


- Don't use sdmx:obsValue as a measure

DON'T


- Don't define measures as sub-properties of sdmx:obsValue

2: Define cubes with multiple measures?

OPEN ISSUES

- Should a cube have one or multiple measures?
- Use cubes with multiple measures only when measures are closely related to a single observational event

3: How to define a unit

DO

- Always define unit for measures

DO


- QUDT vocabulary should be preferred for the units of measure

4: How to define multiple units per measure

Scenario: Publish data about unemployment with two units (percent, count)

OPTION 1 (Publish multiple cubes with one unit)


- Publish two cubes about unemployment with a single unit. :cube1 with qudt:Percent and :cube2 with qudt:Number
- Define the unit at the qb:MeasureProperty

OPTION 2 (Publish one cube with multiple units)


- Publish one cube about unemployment with two units (qudt:Percent, qudt:Number)
- Define the unit at each qb:Observation

5: How to define multiple measures with multiple units

Scenario: Publish data about unemployment (percent, count) and poverty (count)

OPTION 1 (Multi-measure observations)


- Use multiple measures at one observation. For example :obs1 has two measures :unemployment and :poverty

OPTION 2 (Measure dimension)


- Use observations with a single measure
- Use the dimension qb:MeasureType to denote the measure used by the observation

DO

- Use a code list to indicate the set of potential dimension values
- Use rdfs:range to define the type of dimension values
*Exception: a code list should not be defined when the rdfs:range is a data-type e.g. xsd:date, xsd:integer, xsd:boolean
